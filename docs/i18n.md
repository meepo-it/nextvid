# 国际化（i18n）

由 **[Paraglide JS](https://inlang.com/m/gerre34r/library-inlang-paraglideJs)**（`@inlang/paraglide-js`）驱动。Paraglide 把消息 JSON 编译成可 tree-shake 的「一个 message 一个 JS 模块」，**运行时零查表开销**，未使用的字符串会从客户端 bundle 中剔除。

**调用方：** 公共页面（`src/routes/(pages)/*`）、dashboard / settings / admin 路由、邮件模板（`src/mail/templates/*`）、SEO 元数据（`src/lib/seo.ts`）、sitemap（`src/routes/sitemap[.]xml.ts`）、`src/api/feature-requests.ts` 等后台任务。

> ⚠️ **改 i18n 必读：** 任何涉及 i18n 的改动都必须遵循本文档。新加用户可见字符串、加 locale、改语言切换器、发邮件等等，全部按这里规定的模式来。要新增 locale 或字符串时，先看「工作流速查表」。

---

## 支持的 locale

| Locale | 标签     | URL 前缀 | 默认 |
| ------ | -------- | -------- | ---- |
| `en`   | English  | _(无)_   | ✓    |
| `zh`   | 简体中文 | `/zh`    |      |
| `ja`   | 日本語   | `/ja`    |      |

- Base locale：`en`（无 URL 前缀，canonical 指向它）。
- 在 `project.inlang/settings.json` 定义，编译产物 `src/paraglide/runtime.js` 自动同步。
- 当前 `zh` 即「简体中文」。日后要上繁体时直接加 `zh-TW` 作为新 locale（不要改 `zh` 的含义）。
- **加 locale 流程见后文「工作流速查表」**，必须按顺序走完所有步骤，否则测试或运行时会报错。

> **关于繁体中文 (`zh-TW`)：** 当前只支持简体（`zh`）。`src/lib/i18n.ts` 的 `normalizeLocale` 和 `src/components/locale/locale-suggest-banner.tsx` 的 `matchSupportedLocale` **已经做了脚本感知的兜底**：`zh-Hant*` / `zh-TW` / `zh-HK` / `zh-MO` 用户**不会**被错误地推到 `zh`（简体）— 在 `zh-TW` 还没注册时，会回退到 baseLocale 而不是猜。要正式上线繁体，按「工作流速查表 → 加 locale」加 `zh-TW`，脚本兜底自动接管。

## locale 解析顺序

Paraglide 按以下顺序解析当前 locale（见 `src/paraglide/runtime.js`）：

```
1. url              → /zh/pricing, /ja/about
2. cookie           → PARAGLIDE_LOCALE（400 天 maxAge）
3. preferredLanguage → navigator.languages
4. baseLocale       → "en"
```

- **URL 是权威。** `setLocale(l)` 会写 cookie 并跳到带前缀的 URL，绝不只靠 cookie。
- **受保护路由**（`/dashboard`、`/settings`、`/admin`、`/api/*`）不走 URL 前缀（`vite.config.ts` 里的 `protectedPattern`），由 cookie / `preferredLanguage` 决定。
- **SSR：** Vite 插件注册的 `paraglideMiddleware` 会把请求 locale 存入 `AsyncLocalStorage`，所以 server function、loader、邮件渲染里都能直接 `getLocale()`，不用手动传上下文。
- **后台任务/异步邮件：** 见下面「按收件人 locale 渲染邮件」一节，必须用 `withLocale(...)` 显式切换。

## 写消息和用消息

### 1. 定义 key

在 **所有** locale 文件里都加上（当前是 `en.json` / `zh.json` / `ja.json` 三个）。Key 必须是合法 JS 标识符（`snake_case`、小写字母、数字、下划线）。如果 locale 代码里有连字符（例如未来加 `zh-TW`），在 key 中要变成下划线：`language_name_zh_tw`。按业务前缀分组：`nav_`、`auth_`、`pricing_`、`admin_`、`settings_`、`mail_`、`dashboard_`、`feature_requests_`、`contact_`、`about_`、`blog_`、`locale_suggest_`、`language_name_*`、`language_region_*`。

```jsonc
// messages/en.json
{
  "pricing_title": "Simple, transparent pricing",
  "locale_suggest_message": "This page is available in {language}"
}
```

参数用 `{variable}` 语法 — 编译后会变成带类型的函数参数。

### 2. 重新编译

dev / build 时 Vite 插件会自动编译。如果在 dev server 之外手动改了 JSON，重启 dev server 即可。**永远不要手改 `src/paraglide/` 下的文件**，会被覆盖。手动编译命令：

```bash
pnpm exec paraglide-js compile --project ./project.inlang --outdir ./src/paraglide
```

### 3. 在 React 组件里用

```tsx
import * as m from "@/paraglide/messages.js";

export function PricingHero() {
  return (
    <>
      <h1>{m.pricing_title()}</h1>
      <p>{m.locale_suggest_message({ language: "日本語" })}</p>
    </>
  );
}
```

所有 message 都是普通函数 — tree-shake、强类型，**不需要 Provider，也不需要 `useTranslations()` hook**。

### 4. 在路由 `head()` 里用

```tsx
import * as m from "@/paraglide/messages.js";
import { seo } from "@/lib/seo";

export const Route = createFileRoute("/(pages)/pricing")({
  head: () =>
    seo("/pricing", {
      title: `${m.pricing_title()} | ${websiteConfig.metadata?.name}`,
      description: m.pricing_description(),
    }),
});
```

> **规则：** 不允许硬编码任何用户可见字符串。如果某个路由的 `head()` 里出现了字面量 `title` 或 `description`，那就是 bug。受保护路由（`dashboard.tsx` / `settings.tsx` / `admin.tsx`）也必须在 layout 级别提供 `head()`。

### 5. 在 server function / 邮件模板里用

```tsx
import { getLocale } from "@/paraglide/runtime.js";
import * as m from "@/paraglide/messages.js";

// 因为 paraglideMiddleware 在请求开始时设置了 AsyncLocalStorage，
// 这里直接调用 m.* 就会取到当前请求的 locale。
export function FeatureRequestStatusUpdate({ title }: { title: string }) {
  return (
    <EmailLayout>
      <Heading>{m.mail_feature_request_status_update_title()}</Heading>
      <Text>{m.mail_feature_request_status_update_body({ title })}</Text>
    </EmailLayout>
  );
}
```

## 切换语言

```ts
import { setLocale } from "@/paraglide/runtime.js";
import { setUserLocale } from "@/api/user-locale";

// 已登录用户：把 locale 也写到 user 行，便于后台任务发邮件用
void setUserLocale({ data: { locale: "zh" } }).catch(() => {});
setLocale("zh"); // 写 cookie + 跳转 /zh/<当前路径>
```

- **顶部切换器** — `src/components/locale/locale-switcher.tsx`。Dialog 弹窗，3 列网格，旗帜 emoji + 本地名 + 描述 + 当前选中的 checkmark。
- **页脚切换器** — `src/components/locale/footer-locale-switcher.tsx`。行内链接。
- **建议横幅** — `src/components/locale/locale-suggest-banner.tsx`。读 `navigator.languages`，用 `matchSupportedLocale()` 做脚本感知的匹配后弹出建议。简繁互不串：`zh-Hant*` / `zh-TW` / `zh-HK` / `zh-MO` 用户**不会**被建议切到 `zh`（简体），当目标脚本对应的 locale 还没注册时直接放弃建议而不是猜。Dismiss 后写 `PARAGLIDE_LOCALE_DISMISSED` cookie 30 天。

持久化：`PARAGLIDE_LOCALE` cookie（400 天）+ URL 路径前缀 + 已登录用户的 `user.locale` 列。Cookie 优先级高于 `preferredLanguage`，所以用户的显式选择会跨设备/会话保留。

## SEO

集中在 `src/lib/seo.ts` + `src/routes/sitemap[.]xml.ts`：

- **Canonical** — 始终指向 base locale URL（`/pricing`，不是 `/zh/pricing`）。
- **hreflang** — `getHreflangLinks(path)` 为每个支持的 locale 输出一个 `<link rel="alternate" hreflang="…">`，外加一个 `x-default` 指向 base locale。
- **Sitemap** — 每个公共路径只输出一次，用 `<xhtml:link>` 列出所有 locale 备选。**新增页面时记得同步更新 `publicPaths` 数组**，否则被遗漏（参考 commit `ede6efa` 给 `/requests-and-roadmap` 补登录的修复）。
- **`<html lang>`** — `src/routes/__root.tsx` 设置 `<html lang={getLocale()}>`；`src/mail/components/email-layout.tsx` 邮件里也一样。
- **元数据**（`title` / `description` / OG / Twitter）— **必须**通过 `m.*()` 取，绝不硬编码英文。

## 日期 / 数字 / 货币格式化

用 `src/lib/formatter.ts` 里的 helper — 它们默认走 `getLocale()`，可选地接受 `locale` override：

```ts
import { formatPrice, formatDate, formatDateTime, formatBytes } from "@/lib/formatter";

formatPrice(1999, "USD");           // en: "$19"，zh: "US$19"
formatDate(new Date());             // 按 locale 格式化
formatDateTime(new Date());         // 同上 + 24 小时制
formatBytes(1536);                  // 小数分隔符跟随 locale

// override（比如给某个特定用户发邮件）：
formatPrice(1999, "USD", "ja");
```

如果一定要在 helper 之外格式化，**必须**显式传 `getLocale()`，**绝不**写死 `'en-US'`：

```ts
import { getLocale } from "@/paraglide/runtime.js";
new Intl.NumberFormat(getLocale(), { style: "currency", currency }).format(price / 100);
```

## 按收件人 locale 渲染邮件（重要）

后台任务（webhook、status update、批量通知等）发出的邮件**应该按收件人的语言渲染**，不是当前请求的语言。否则管理员用中文操作时，所有日文用户都会收到中文邮件。

### 数据来源

- `user.locale`（`src/db/auth.schema.ts`）— 用户在 locale 切换器里挑选的语言（由 `setUserLocale` server function 写入）。
- `feature_request.locale`（`src/db/app.schema.ts`）— 提交时记录的语言（`createFeatureRequest` 自动调 `getLocale()`）。

### 用法

用 `src/lib/i18n.ts` 提供的 `withLocale(locale, fn)` helper 包住 `sendEmail` 调用。它通过 Paraglide 的 `serverAsyncLocalStorage` 临时切换 `getLocale()`，**不会影响其他并发请求**。

```ts
import { withLocale } from "@/lib/i18n";
import { sendEmail } from "@/mail";
import { getLocale } from "@/paraglide/runtime.js";

await Promise.allSettled(
  recipients.map((r) =>
    withLocale(r.locale ?? getLocale(), async () => {
      // r.locale 没有就退回当前请求 locale
      // 在这个回调里调 m.* 都会按 r.locale 解析
      const oldLabel = statusLabel(current.status);
      const newLabel = statusLabel(data.status);
      return sendEmail({
        to: r.email,
        template: "featureRequestStatusUpdate",
        context: { title: current.title, oldStatus: oldLabel, newStatus: newLabel },
      });
    })
  )
);
```

### 规则

- **任何**写「向多个用户群发」或「webhook 触发邮件」的代码，**必须**用 `withLocale` 包裹。
- 把状态/枚举的「英文标签」当字符串存到模板上下文里 = bug。要在 `withLocale` 闭包内调本地化函数（比如 `statusLabel(...)`）再传入。
- 用户可能没有 `locale`（早期数据 / 未登录就提交）。用 `r.locale ?? getLocale()` 兜底，`withLocale` 内部还会用 `normalizeLocale` 二次清洗。

## 动态 / 用户生成内容

数据库里的 user-generated 内容（feature requests、contact messages、blog 文章）按**单一语言**存（作者提交时的语言）。`feature_request` 有 `locale` 字段记录是哪种语言，但**正文本身不做翻译**。

规则：

- 内容外围的 UI 文案（标签、按钮、空状态、status badge）**必须** `m.*()` 本地化。
- 内容正文原样渲染。
- 如果某个新功能确实需要多语言正文，加 `locale` 列或 JSONB `translations` 字段，**不要复制行**。

## 第三方集成

- **Stripe Checkout / Customer Portal** — `src/api/payment.ts` 在调 `createCheckout` / `createCustomerPortal` 时自动注入 `getLocale()`，`src/payment/provider/stripe.ts` 通过 `toStripeLocale()` 转一道再传给 Stripe。当前 `zh` 直接对应 Stripe 的 `zh`（简体），不需要映射；新增的 locale 如果 Stripe 名字不一样（例如未来加 `zh-TW`），去 `toStripeLocale()` 的 `map` 里加一行，否则 Stripe 会回退到 `auto`。参考 [Stripe locale 列表](https://docs.stripe.com/payments/checkout/customization/appearance)。
- **Resend** — 内容通过模板本地化；主题通过 `src/mail/render.ts` 的 `EmailSubjects` 映射本地化。后台批量发邮件**必须**结合 `withLocale`，见上一节。

## 测试 i18n

`src/__tests__/i18n.test.ts` 会读 `project.inlang/settings.json` 自动遍历每个非 base locale，对以下规则做断言（**新增 locale 不需要改测试**）：

- key 集合与 base locale 完全一致（不缺、不多）。
- key 数量相同。
- 没有空值。
- 占位符变量（`{name}`）每个 key 都和 base locale 匹配。
- **「逐字 copy」防呆：** 若 ≥ 20% 的值与 base locale 字节相同，测试失败（防止「`cp en.json ja.json` 然后忘了翻译」）。

提交任何 `messages/*.json` 改动前都要跑：

```bash
pnpm exec vitest run src/__tests__/i18n.test.ts
```

## 场景清单（防遗漏）

每条场景列出需要动到的所有 i18n 触点。**改/加东西时按这个清单逐项核对**，否则容易漏掉 sitemap、hreflang、邮件、测试中的某一处。

### ① 加一个用户可见字符串

- [ ] `messages/en.json` 加 key（snake_case，按业务前缀）
- [ ] `messages/zh.json` 加翻译（不能照抄英文）
- [ ] `messages/ja.json` 加翻译
- [ ] 如果有占位符，三处 `{var}` 名字必须一致
- [ ] 在使用方 `import * as m from '@/paraglide/messages.js'` 调 `m.your_key()`
- [ ] 跑 `pnpm exec vitest run src/__tests__/i18n.test.ts`
- [ ] **禁止**直接在 JSX/字符串里写中英文

### ② 新加一个公共页面（`src/routes/(pages)/<name>.tsx`）

- [ ] 文案全部走 `m.*()`（标题、副标题、按钮、空状态、错误提示）
- [ ] 路由 `head` 用 `seo(path, { title: m.x(), description: m.y() })`
- [ ] **`src/routes/sitemap[.]xml.ts` 的 `publicPaths` 加新路径**（否则 sitemap 缺失 + hreflang 不生成）
- [ ] **`vite.config.ts` 的 `urlPatterns` 加 `publicPattern('/your-path')`**（否则 `/zh/your-path` 不会生成）
- [ ] 如果是动态路由（`/blog/:slug`），同样加 `publicPattern('/blog/:slug')`
- [ ] 跑一次 dev server 确认 `/zh/<path>`、`/ja/<path>` 都能访问

### ③ 新加一个受保护页面（dashboard / settings / admin 子路由）

- [ ] 文案全部走 `m.*()`
- [ ] **不需要**改 `vite.config.ts`（受保护路由由 `protectedPattern` 通配，无 locale 前缀）
- [ ] **不需要**进 sitemap
- [ ] 子页面如果有自己的 `head()`，也用 `seo()` + `m.*()`
- [ ] layout 级别已经有 `head()`（dashboard / settings / admin），新建子页面不强制重写

### ④ 新加一个 React 组件

- [ ] 所有 visible 文案走 `m.*()`，包括 `aria-label`、`placeholder`、`title` 属性
- [ ] 不要把状态/枚举的英文字面量当文案传 — 用 `m.*()` 解析后再渲染
- [ ] toast / dialog / tooltip 文案同样要 i18n
- [ ] 表单校验错误信息走 `m.*()`（zod schema 用 `.refine()` 时 message 也是）
- [ ] 日期 / 数字 / 货币用 `src/lib/formatter.ts` 的 helper（默认走 `getLocale()`）

### ⑤ 新加一个邮件模板

- [ ] 在 `src/mail/templates/` 创建 `<name>.tsx`，文案全部 `m.mail_<name>_*()` 系列
- [ ] 三个 message 文件都加对应 key（subject、title、body、button、greeting 等）
- [ ] 在 `src/mail/types.ts` 的 `EmailTemplate` 联合类型加 `'<name>'`
- [ ] 在 `src/mail/render.ts` 的 `EmailTemplates` 注册组件
- [ ] 在 `src/mail/render.ts` 的 `EmailSubjects` 注册 `m.mail_<name>_subject`
- [ ] **后台/批量发送场景**：调用方必须用 `withLocale(recipient.locale ?? getLocale(), () => sendEmail(...))` 包裹（见场景 ⑦）

### ⑥ 新加一个 server function（`src/api/*.ts`）

- [ ] 内部用到的「需要本地化的字符串」走 `m.*()`，**不要**把英文写死后返回给前端再翻译
- [ ] 抛出 `Error` 的 message 看场景：仅 dev 调试用就英文；要展示给用户就 `m.*()`
- [ ] 校验失败 toast 等用户可见的反馈走 `m.*()`
- [ ] 如果落库的字段记录用户语言（feature_request / 用户内容），插入时写 `locale: getLocale()`
- [ ] 如果写完之后会**异步**给某个用户发邮件（webhook、cron、状态变更），按场景 ⑦ 处理

### ⑦ 后台任务 / 异步邮件 / 群发

- [ ] **必须**用 `withLocale(recipient.locale ?? getLocale(), () => sendEmail(...))` 包裹 `sendEmail`

- [ ] `Promise.allSettled(recipients.map(r => withLocale(r.locale ?? getLocale(), async () => { ... })))` 是标准模式

### ⑧ DB 模式改动（feature_request / user / 任何记录用户语言的表）

- [ ] schema 用 `text('locale')`（nullable 表示「未指明，回退到请求 locale」），或 `.notNull().default('en')`
- [ ] insert 时写 `locale: getLocale()`

- [ ] migration 记得部署到 dev / preview / prod

### ⑨ 加一个新 locale（例如未来加 `zh-TW`）

- [ ] 三个 message 文件加 `language_name_zh_tw` + `language_region_zh_tw`（key 中连字符变下划线）

## 已知遗留

- **DB 内容是单语言的。** Feature requests / contact messages / blog 文章正文都按提交时语言存。`feature_request.locale` 列只记录是哪种语言，并不存翻译。如需真正多语言正文，建议加 `locale` 列或 JSONB `translations` 字段，**不要复制行**。
- **`zh-Hant` 单独 locale 还没加。** 当前繁中用户在建议横幅里**不会**被推荐切到简中（已修），但也没有专属的繁中 locale。要加的话按「工作流速查表」走流程，并扩展 `locale-suggest-banner.tsx` 的繁中检测。
